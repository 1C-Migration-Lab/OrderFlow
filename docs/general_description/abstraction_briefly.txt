Ниже перечислю **уровни абстракции**, которые можно (теоретически) построить по описанной **мини-конфигурации «Мини Заказы»**.  
Это та самая последовательность от AST до «бизнес-модели», о которой мы говорили ранее:

1. **Source Code**  
   - У нас есть «сырой» текст модулей 1С: «Модуль объекта ЗаказПокупателя», «Модуль менеджера», модули форм и т. п.

2. **Lexical Analysis / Токены**  
   - Можно запустить лексер и получить список токенов (ключевые слова, идентификаторы, операторы и т. д.) — пригодится для последующего парсинга.

3. **Parse Tree / Дерево разбора**  
   - На основе токенов построить дерево разбора, где будут узлы типа «ProcedureDeclaration», «IfStatement», «VariableDeclaration» и т. д.

4. **AST / Абстрактное синтаксическое дерево**  
   - Из дерева разбора сформировать AST, убрав «лишние» служебные узлы.  
   - В итоговом AST будут объекты вроде «ПроцедураПриЗаписи», «ПроцедураПриПроведении», «ЦиклПоТабЧасти» и т. д.

5. **Code Model / IR (Промежуточное представление кода)**  
   - Объединить AST по модулям и объектам (документ «ЗаказПокупателя», регистр «ЗаказыПоКонтрагентам», справочники «Номенклатура» и «Контрагенты»).  
   - Описать процедуры, функции, их параметры, где и как они вызываются.

6. **Dependency Graph / Граф зависимостей**  
   - Построить связи (например: «Модуль объекта ЗаказПокупателя» вызывает «Движения.ЗаказыПоКонтрагентам», читает/записывает значения реквизитов).  
   - Определить кто вызывает что, какие общие модули задействованы (если бы они были), где хранятся данные.

7. **Архитектурный / Функциональный уровень**  
   - Здесь видим структуру конфигурации: 2 справочника, 1 документ, 1 регистр, формы.  
   - Понимаем, что «это» — подсистема «Мини-Продажи» (условно).

8. **Domain Model / Предметная модель**  
   - В терминах бизнеса: есть сущность «Товар» (Номенклатура), «Клиент» (Контрагент), «Заказ», «Позиции заказа» и «Регистр остатков/учёта заказов».  
   - Здесь уже описываем, что «Заказ» — это не просто документ, а бизнес-объект, у которого есть сумма, список товаров и т. д.

9. **Business Processes / Бизнес-процессы**  
   - Минимальный сценарий: «Создание заказа», «Заполнение ТЧ», «Проведение заказа» (запись в регистр).  
   - Можно описать простую цепочку: *Номенклатура -> Создание Заказа -> Проведение -> Отчёт по заказам*.

10. **Business Model / Бизнес-модель**  
   - Совсем верхний уровень: «В системе есть модуль продаж, позволяющий регистрировать заказы клиентов и видеть общую сумму заказов по каждому контрагенту».  
   - Это описание без технических деталей 1С, понятное для бизнес-стейкхолдеров.

Все эти уровни, **от лексеров/парсеров (AST)** до **доменной и бизнес-модели**, могут быть (в теории) применены даже к крошечной конфигурации, чтобы продемонстрировать полный путь от сырого кода до смысла для бизнеса.
